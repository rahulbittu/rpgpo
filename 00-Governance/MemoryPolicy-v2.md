# RPGPO Memory Policy v2

## Purpose

RPGPO memory exists to improve continuity, quality, context retention, and long-term leverage.

Memory is not a dumping ground.
Memory is not a random archive.
Memory is a governed system for preserving what should persist and excluding what should not.

The goal of memory is to help RPGPO become more useful over time without becoming contaminated, bloated, invasive, or careless.

---

## Core Principles

### 1. Memory must earn its place
Not everything should be remembered.
Only information with repeated strategic, operational, or creative value should persist.

### 2. Memory must be scoped
Memory should belong to the correct layer and domain.
Sensitive memory must not leak into unrelated work.

### 3. Memory must support action
Stored memory should improve:
- decisions
- execution speed
- quality of outputs
- consistency
- leverage
- personalization where appropriate

### 4. Memory must not silently expand
New memory domains must be explicitly introduced, not casually absorbed.

### 5. Memory must be reviewable
Important persistent memory should be inspectable, editable, and removable.

---

## Memory Layers

## Layer 1: Constitutional Memory
This is permanent and shared across all RPGPO offices.

Includes:
- constitution
- core values
- board roles
- approval rules
- logging standards
- access principles
- identity of RPGPO

This layer is inherited by every child office.

---

## Layer 2: Rahul Core Memory
This stores long-lived information about Rahul’s working style, preferences, standards, priorities, and long-term goals.

Examples:
- communication preferences
- ambition level
- product philosophy
- work style
- preferred output style
- recurring priorities
- risk tolerance by category
- standards for quality and tone

This layer should be stable and selectively updated.

---

## Layer 3: Domain Memory
Each major domain maintains its own memory.

Examples:
- TopRanker memory
- Founder2Founder memory
- Career Engine memory
- Wealth Research memory
- Creative Office memory

Each domain memory may include:
- domain-specific goals
- current state
- recurring workflows
- key assets
- style rules
- constraints
- active decisions
- known history relevant to that domain

Rules:
- domain memory should not freely spill into other domains
- sharing across domains should be deliberate

---

## Layer 4: Active Operating Memory
This stores current tasks, ongoing initiatives, today’s priorities, and near-term working context.

Examples:
- active sprint tasks
- pending approvals
- currently researched questions
- current job shortlist
- open growth experiments

This layer is temporary and should be cleaned regularly.

---

## Layer 5: Sensitive Scoped Memory
This is restricted memory for specific sensitive items.

Examples:
- specific shared email threads
- selected private documents
- selected financial notes
- sensitive negotiations
- privileged communications
- personal logistics explicitly shared

Rules:
- access should be narrow
- retrieval should be intentional
- these items should not appear in general summaries unless relevant and allowed

---

## What Should Be Remembered

RPGPO should remember:
- long-term goals
- preferences that affect recurring outputs
- project structures
- product decisions
- repeated operating rules
- important style/tone decisions
- standing constraints
- recurring workflows
- strategic lessons learned
- reusable insights

---

## What Should Not Be Remembered by Default

RPGPO should not remember by default:
- random temporary chatter
- low-value one-off details
- irrelevant browsing noise
- emotionally loaded material without durable value
- sensitive personal details beyond approved scope
- entire inboxes or unrelated threads
- transient drafts that are no longer relevant

---

## Memory Update Rules

Before storing something persistently, ask:

1. Will this matter again?
2. Does this improve future work quality?
3. Does it belong in a specific domain?
4. Is it sensitive?
5. Should it be summarized instead of stored raw?

If the answer is weak, do not store it.

---

## Memory Quality Standard

Good memory is:
- concise
- structured
- reusable
- relevant
- scoped correctly
- easy to review

Bad memory is:
- bloated
- redundant
- emotionally noisy
- sensitive without justification
- disconnected from action

---

## Retrieval Rules

When retrieving memory:
- retrieve the narrowest useful layer first
- prefer domain memory over broad memory if the task is domain-specific
- do not expose sensitive scoped memory unless required
- clearly distinguish between retrieved memory and fresh inference

---

## Memory Governance

Every child office must define:
- what it can remember
- where that memory lives
- what is shared with the core system
- what is restricted
- how often it is reviewed

---

## Review Cadence

Memory should be reviewed:
- weekly for active operating memory
- monthly for domain memory
- quarterly for core memory
- immediately for sensitive memory if scope changes

---

## Final Rule

RPGPO memory should increase usefulness without increasing chaos.
The system should remember enough to become sharper, but never so much that it becomes sloppy, invasive, or contaminated.
